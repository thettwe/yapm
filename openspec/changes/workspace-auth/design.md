# workspace-auth ŌĆö design

## Context

foundation shipped a running Zero pipeline over a single `workspace` row with `AuthContext = {userID:'anonymous', role:'member'}` hard-coded by `resolveAnonymousContext`, and query/mutator authorization already filter-based (`denyAll` = empty `or()`) and driven by `ctx` rather than args. Every seam this change needs is already physical: `packages/schema/src/zero/context.ts` owns the role model and the `DefaultTypes` context, `apps/server/src/zero/context.ts` owns the `ResolveAuthContext` function foundation explicitly left for us to replace, and the routes already pass a verified `userID` to `handleQueryRequest`/`handleMutateRequest`.

The stack is fixed by TECHSTACK.md and the reference notes: better-auth 1.6 on its **native Kysely layer** (the only layer where `getMigrations()` runs from app code at boot ŌĆö `reference/kysely-stack.md` ┬¦5.4), sharing the one `pg.Pool`; Zero 1.8 synced queries with the multi-tenant membership pattern (`reference/zero.md` ┬¦4.5); Kysely 0.28.17 pinned with a hand-written `DB` interface and a CI drift test. This change must not add a container, must keep the viewer role free/unlimited, and must keep all ZQL and mutators inside `packages/schema`.

## Goals / Non-Goals

**Goals:**
- Real identity (better-auth) and real authorization (workspace role + team membership) replacing the anonymous placeholder.
- A user syncs only rows they can see; permissions live in the synced-query definitions, expressed with joins, denied by empty result.
- Email/password + GitHub OAuth + OIDC/SSO all work and are all free; viewer is free and unlimited.
- A fresh instance is usable in minutes with no SMTP and no manual admin creation.
- The three-container contract and the sub-100ms local-read budget are preserved.

**Non-Goals:**
- Issues / work entities (issue-core), GitHub data sync (github-sync), billing, multiple workspaces, per-team roles, SCIM, audit logs.

## Decisions

1. **Identity (better-auth `user`) is separate from authorization (`workspace_member`).** better-auth owns `user`/`session`/`account`/`verification`/`jwks` (camelCase, `text` PKs, singular names ŌĆö ┬¦5.4). We add `workspace_member(workspace_id, user_id, role, ŌĆ”)` as the access edge: presence of a row = access to the instance; `role` = capability. Signing up creates a `user` but **not** membership. *Alternative rejected:* better-auth's `organization` plugin ŌĆö it models many orgs/teams with its own role tables, duplicating our single-workspace graph and pulling role logic out of `packages/schema` where the mutators must live.

2. **Workspace-level roles, team membership orthogonal.** `workspace_member.role Ōłł {admin, member, viewer}` is the single source of capability; `team_membership(team_id, user_id)` is a plain edge with no per-team role. Role gates *what you can do*; team membership gates *which teams' work data you sync*. This keeps the permission model two-axis and simple, and matches the roadmap's flat role list. *Alternative rejected:* per-team roles (Jira-style scheme labyrinth ŌĆö VISION "not a config sandbox").

3. **`ctx = {userID, role}` where `role` is the workspace role or `null` for an authenticated non-member.** The server resolves it once per request by verifying the session/JWT to a `userID`, then looking up `workspace_member.role`. Team membership is *not* precomputed into `ctx` (it is multi-valued) ŌĆö queries check it with a `whereExists('team' ŌåÆ 'members', m => m.where('userID', ctx.userID))` subquery keyed off `ctx`, never off args (┬¦4.5). `role: null` ŌćÆ every query `denyAll`s and every mutator throws `not_authorized`.

4. **Row-level read permissions, per entity:**
   | Entity | Who reads which rows | Who writes |
   |---|---|---|
   | `workspace` (1 row) | any member (role ŌēĀ null) | admin (rename, settings) |
   | `user` profile | any member reads all member profiles (assignee/mention pickers need them) | a user edits only their own |
   | `workspace_member` | any member reads all (see the roster) | admin (invite/role/remove); a member may remove *itself* (leave) |
   | `team` | any member reads all teams | admin (create/rename/archive) |
   | `team_membership` | any member reads all | admin manages any; a member may add/remove *itself* on any team it can see (self-serve join/leave) |
   | `invite` | **admin only** | admin (create/revoke); accept is token-driven, not a synced write |
   Denials are empty queries; mutators check auth **before** existence so a private row's existence never leaks (┬¦5.4).

5. **First-admin bootstrap is first-user-wins, advisory-locked, with an env override.** On completed sign-in, if `workspace_member` has zero rows the caller is inserted as `admin` under `pg_advisory_xact_lock` (same pattern foundation used for seeding ŌĆö design decision 22). If `YAPM_BOOTSTRAP_ADMIN_EMAIL` is set, only a caller whose verified email matches becomes the bootstrap admin; others wait for an invite. Default is zero-config (first user becomes admin) to honor the "under 5 minutes" north star. *Alternative rejected:* a CLI/env-only admin seed (extra step, breaks minutes-to-first-team).

6. **Membership is invite-gated after bootstrap; registration stays open.** An authenticated user with no membership and no accepted invite is a non-member: they see nothing (denyAll) and land on an access-gate screen. Access is granted only by accepting an invite (email or link) or by an admin adding them. "Open join" is expressed as a shareable link an admin chooses to distribute ŌĆö not a separate global toggle. *Alternative rejected:* auto-membership for anyone who can authenticate (unsafe for an internet-reachable private instance).

7. **Invites: one `invite` table, two shapes.** `invite(id, workspace_id, team_id?, email?, role, token, created_by, expires_at, revoked_at?, created_at)`. `email = null` ŌćÆ reusable shareable link (valid until expiry/revoke); `email` present ŌćÆ single-use, bound to that email, revoked on accept. Accepting inserts `workspace_member` (and `team_membership` if `team_id` set). The token link works without SMTP (always copyable); SMTP, when configured, additionally emails it. *Alternative rejected:* separate `invite` and `invite_link` tables (duplicated accept logic).

8. **Email verification off by default.** `emailAndPassword.requireEmailVerification = false`; a fresh self-host needs no SMTP to onboard. When SMTP env is present an admin can require it. Bias-to-off is the documented default per the assignment.

9. **Session 30 days rolling; Zero JWT 1 hour with refresh.** better-auth session `expiresIn: 30d, updateAge: 1d` (self-host convenience ŌĆö long-lived, refreshed on use). The `jwt()` plugin issues a 1-hour token for zero-cache (its 15-minute default is too short for a long-lived sync socket ŌĆö ┬¦5.5); the web client fetches it from `/api/zero/token` and re-fetches on a 401/403 from the sync endpoints, calling `zero.connection.connect({auth})`. `bearer()` + `jwt()` both enabled: bearer for our own API, JWT for zero-cache.

10. **Server derives `ctx` by verifying the JWT locally against the JWKS.** The `/query` and `/mutate` routes read the bearer JWT zero-cache forwards, verify it against better-auth's `jwks` table (issuer/audience), extract `sub` as `userID`, then look up the role. The `X-Api-Key` gate between zero-cache and the endpoints (foundation decision 47/54) stays as defense-in-depth. *Alternative rejected:* calling `auth.api.getSession` on every sync request (a DB round-trip per query batch; JWKS verification is local and cache-friendly).

11. **`WORKSPACE_ROLES` becomes `['admin','member','viewer']`.** foundation used `owner`; the roadmap and this change standardize on `admin`. `canWrite` stays `role !== 'viewer'` (and non-null); `canManage` (new) is `role === 'admin'`. This is a one-line enum change plus the migration mapping any existing `owner` seed to `admin`.

## Risks / Trade-offs

- **`jose` v5/v6 collision** (Zero wants 5, better-auth wants 6 ŌĆö ┬¦0/┬¦9) ŌåÆ pnpm's isolated store keeps both side by side (foundation verified `jose@5` and `jose@6` coexisting once better-auth lands); pin via catalog and `pnpm peers check` in CI. Never hoist a single jose to the root.
- **Bootstrap race across replicas** ŌåÆ the first-admin insert is wrapped in `pg_advisory_xact_lock` + `where not exists`, the exact pattern foundation proved with three concurrent booters producing one row.
- **better-auth `getMigrations()` is unguarded (no advisory lock, not transactional ŌĆö ┬¦5.4)** ŌåÆ run it from the single app boot path after the Kysely `Migrator` (which *is* advisory-locked); accept "relation already exists" as idempotent on concurrent boots, or gate it behind the same lock. Document in tasks.
- **JWT expiry mid-sync stalls writes** ŌåÆ 1-hour token + client refresh on 401/403; the connection indicator (foundation) already surfaces `needs-auth`.
- **Non-member confusion ("I signed in but see nothing")** ŌåÆ an explicit access-gate screen, not a blank/denied app, telling the user they need an invite.
- **Drift test surface grows** ŌåÆ the ┬¦8 drift test now covers five new tables plus better-auth's tables; better-auth tables are excluded from the Zero-schema half (Zero does not replicate them for clients) but the `user` table *is* in the Zero schema (read-only for members), so its columns are drift-checked.

## Migration Plan

Forward-only Kysely migration `0002_workspace_auth` (or split per entity) adds `workspace_member`, `team`, `team_membership`, `invite`, extends `workspace`, and remaps role `owner`ŌåÆ`admin` in any seeded data. better-auth tables are created by `getMigrations()` at first boot. Rollback within v1 pre-release is a DB reset (`pnpm dev:reset`); no production data exists yet. Boot order: Kysely `Migrator` ŌåÆ better-auth `getMigrations()` ŌåÆ seed workspace ŌåÆ serve; zero-cache starts after the app is healthy (unchanged).

## Decisions made during implementation

<!-- Populated during the apply phase per CLAUDE.md's decide-log-continue policy.
     The product decisions the Propose phase was asked to make and log up front: -->

- **Entity/membership/role model**: workspace-level roles (`workspace_member.role Ōłł admin/member/viewer`) with an orthogonal `team_membership` edge (no per-team role). Rationale in Decisions #1ŌĆō#2.
- **Email verification default**: OFF (`requireEmailVerification: false`) so self-host needs no SMTP. Decision #8.
- **Invite mechanism**: BOTH single-use email invites and reusable shareable links, one `invite` table. Decision #7.
- **First-admin bootstrap**: first authenticated user becomes admin (advisory-locked), optional `YAPM_BOOTSTRAP_ADMIN_EMAIL` override. Decision #5.
- **Session/JWT lifetimes**: session 30d rolling (updateAge 1d); Zero JWT 1h with client refresh on 401/403. Decision #9.
- **OIDC/SSO**: kept in this change and free/ungated (`@better-auth/sso`). Proposal + authentication spec.

### Schema phase (data model in `packages/schema`)

- **No app-level FK to better-auth's `user`.** Boot order is Kysely `Migrator` ŌåÆ better-auth `getMigrations()` (Migration Plan above), so at the moment migration `0002_workspace_auth` runs the `user` table does not exist yet. A DB-level `references "user"(id)` would fail. `workspace_member.user_id`, `team_membership.user_id`, and `invite.created_by` are therefore plain `text` columns with no hard FK; referential integrity to identity is enforced at the mutator/app layer. FKs among our own tables (member/team/team_membership/invite ŌåÆ workspace/team) are kept, since those are all created inside `0002`. This makes the migration order-independent from better-auth.
- **`workspace` table left unchanged.** Task 2.2 says "extending `workspace`," but neither the proposal nor any spec names a new `workspace` column, and VISION forbids a config sandbox. `workspace` keeps its foundation shape (`id, name, created_at, updated_at`) and satisfies "extend" by being the instance root the new FKs hang off. A future change can add columns when a concrete need appears.
- **Roles are `text` + `CHECK (role in ('admin','member','viewer'))`, not a Postgres `enum` type.** The Zero column is `enumeration<WorkspaceRole>()` (whose runtime `type` is `"string"`, verified from the 1.8 `.d.ts`), so it maps cleanly through the drift test's `text ŌåÆ string` rule. A real Postgres enum type would surface as an unmapped `dataType` in the drift guard and add a migration-fragile type. Client typing is preserved via the Zero enumeration; validity is enforced by the CHECK and by mutator-side Zod (later phase).
- **`owner ŌåÆ admin` remap is a no-op.** Foundation persisted no role data ŌĆö `owner` existed only in `WORKSPACE_ROLES` and tests. `workspace_member` is created fresh in `0002`, so there is nothing to remap; the enum change is code-only (`WORKSPACE_ROLES`, updated tests). The CHECK constraint enforces the new set going forward.
- **The drift test provisions better-auth's `user` table itself.** `packages/schema` must not import the server's better-auth config (packages never import apps). To still drift-check the read-surface `user` interface and Zero table, the test creates `"user"` from the verified `getMigrations()` DDL (reference/kysely-stack.md ┬¦5.4) in `beforeAll`, then asserts the hand-written shapes match it. The full `user` column set is included in the Zero schema so the drift guard's bidirectional check (no DB column missing from the Zero schema) stays intact.
- **`bootstrapFirstAdmin` is a data-layer primitive in `db/seed.ts`.** First-admin promotion is a Postgres operation (advisory-locked insert-if-no-members honoring `requiredEmail`), so it lives beside `seedWorkspace` and is unit-testable without the server. The server phase (task 4.5) calls it from the completed-sign-in hook, minting the `workspace_member` id via `newId()` at that call site (constraint #4). Verified against real Postgres: concurrent first sign-ins yield exactly one `admin`, and the `requiredEmail` gate rejects non-matching callers.
- **`team_membership` has a client-minted `id` PK plus a `unique(team_id, user_id)`,** rather than a compound PK. This matches constraint #4 (client-minted UUIDv7 PKs at the call site) and the repo's single-`id` convention, while the unique constraint still guarantees one membership row per (team, user).

### Auth phase (better-auth in `apps/server`)

- **`better-auth` and `@better-auth/sso` are added only to `apps/server`, not to `packages/schema`.** Task 1.1 suggested `packages/schema` for "auth-adjacent DB types," but the Schema phase already decided the drift test reproduces better-auth's `user` DDL itself (packages never import apps), and `packages/schema` has no code that imports better-auth. Adding it there would be an unjustified, unused dependency. The `user` read surface stays a hand-written interface. `jose@^6` is added to the catalog and `apps/server` for local JWKS verification (better-auth wants jose 6, Zero wants jose 5; pnpm's isolated store keeps both ŌĆö verified: `apps/server` resolves 6.2.4, Zero 5.10.0, `pnpm peers check` shows no jose issue).
- **Per-request role resolution, not role-in-token.** `verifySyncToken` returns only the verified `sub`; the role is looked up per request via `lookupWorkspaceRole` (design #3/#10). This reflects role/membership changes immediately (a removed user loses access on the next transform) instead of staying stale until a 1-hour token refresh. The `/api/zero/token` response additionally returns the current role so the *client* can build its optimistic `context` ŌĆö that value grants no authority (the server re-verifies and re-resolves independently).
- **JWKS is fetched over loopback (`127.0.0.1:PORT`), not the public `BETTER_AUTH_URL`.** Token verification must not depend on the container being able to reach its own external URL. Tokens are still verified against the `BETTER_AUTH_URL` issuer/audience; only the key-fetch host is loopback. Verified against real Postgres with `HOST=0.0.0.0`.
- **First-admin bootstrap runs in the `/api/zero/token` handler** (the first authenticated call the SPA makes after sign-in), calling the Schema-phase `bootstrapFirstAdmin` primitive with a call-site-minted id. This avoids depending on an unverified better-auth hook API while still promoting the first user (advisory-locked, `YAPM_BOOTSTRAP_ADMIN_EMAIL`-gated). Verified: the first email sign-up yields one `admin` `workspace_member` row.
- **Invite acceptance is a REST endpoint (`POST /api/invites/accept`), not a synced mutator** (design #4/#7: "accept is token-driven, not a synced write"). The DB logic lives in `packages/schema/src/db/invite.ts` (`acceptInvite`), beside `bootstrapFirstAdmin`, so it is unit-tested against real Postgres without the server; the thin Hono handler maps refusals to 404/403/409.
- **Optional env vars treat empty strings as unset.** `optionalString` preprocesses `""`/whitespace to `undefined` so a docker-compose `${GITHUB_CLIENT_ID:-}` (empty) disables the provider instead of failing Zod validation and crashing boot. Verified: booting with `GITHUB_CLIENT_ID=""` etc. starts cleanly.
- **SSO/OIDC providers are registered at runtime via the `@better-auth/sso` plugin, so there is no static OIDC env.** The plugin is enabled unconditionally (free, ungated); specific IdPs are added through its API/DB, matching "a method whose credentials are unset is simply unavailable." No fabricated static-OIDC config was added.
- **The `resolveSessionContext` swap tightens sync to authenticated members only; the non-member `denyAll` for the existing `workspace.current` query is deferred to the Sync/permissions phase (task 3.2).** In the Auth phase, absent/invalid credentials already yield no context (`denyAll`, verified). An authenticated non-member resolves to `role: null`; `workspace.current` still uses `canRead` (true for any context), so it would show the single workspace name to a signed-in non-member until 3.2 rewrites the queries with role-`null` denial. No sensitive rows exist yet, and identity/authority resolution ŌĆö the Auth-phase deliverable ŌĆö is correct.
- **Session middleware is scoped to the app's own authenticated endpoints** (`/api/zero/token`, `/api/invites/accept`) via a `requireSession` middleware, not applied globally, so `/query`, `/mutate`, and the health probes never pay a `getSession` DB round-trip.
### Sync/permissions phase (row-level queries and role-authorized mutators in `packages/schema`)

- **Membership is gated by `isMember` (role ŌēĀ null), and `canRead` now delegates to it.** The Auth phase left `workspace.current` on the looser `canRead` (any authenticated context) and deferred role-`null` denial to task 3.2. This phase adds `isMember(ctx) = ctx !== undefined && ctx.role !== null` and redefines `canRead` to delegate to it ŌĆö every workspace-scoped row requires membership, so an authenticated non-member now `denyAll`s on every query. The export surface is unchanged (`canRead` still exported); only its gate tightened. Verified: `members/users/teams/workspace.current` return an empty `or()` where-clause for `role:null` and `undefined`; `invites.all` denies everyone but an admin.
- **Workspace rename is admin-only (`canManage`), correcting the Auth-phase `canWrite` gate.** The workspace-membership spec says "Only an `admin` SHALL rename the workspace"; the foundation/Auth-phase mutator used `canWrite` (member+). This phase tightens `assertRenameWorkspaceAllowed` to `canManage` and updates the mutator test (a member is now rejected). No other caller depended on the looser gate.
- **Invite `accept` stays the token-driven REST endpoint (task 4.6), not a synced mutator, despite task 3.3 listing "accept."** Invite rows are admin-only readable (design #4/#7), and a client-side mutator's `tx.run` only sees already-synced rows (reference/zero.md ┬¦5.2) ŌĆö so a non-member invitee could never read the invite to validate it optimistically. Accept must run server-side against the token, which is exactly the existing `POST /api/invites/accept` ŌåÆ `acceptInvite` path (already unit-tested against real Postgres). Adding a synced accept mutator would either leak invite rows to non-members or be a client no-op. No `decline` mutator was added: there is no spec scenario for it, and declining a token invite has the same non-member-can't-see-it constraint (a future token-driven REST revoke-on-decline can cover it if a need appears).
- **Self-serve join/leave and admin roster management are unified into `team.addMember` / `team.removeMember`.** Rather than four near-identical mutators, `addMember({id, teamId, userId})` allows the call when `userId === ctx.userID` (any member, including a viewer ŌĆö joining grants no write power) or when `canManage` (admin adds anyone); `removeMember({id})` allows it for one's own membership (leave) or for an admin (remove anyone). This satisfies both the teams spec's self-serve join/leave and admin-manages-any requirements with one code path each. Archived/nonexistent teams are refused (`team.archivedAt != null`), and a viewer's join is accepted while every write mutator still rejects the viewer.
- **`member.remove` likewise covers admin-remove and self-leave in one mutator**, with `assertNotLastAdmin` applied whenever the target is an admin (so the last admin can neither be removed by an admin nor leave voluntarily ŌĆö the instance is never left unadministered). `member.changeRole` is admin-only and runs the same last-admin guard before demoting the sole admin.
- **`teamScoped(query, ctx)` is a forward-looking `whereExists` helper for issue-core work data.** It denies non-members (`denyAll`) and otherwise scopes a query to rows whose `team` the caller belongs to via `whereExists('team', t => t.whereExists('members', m => m.where('userId', ctx.userID)))` ŌĆö the ┬¦4.5 membership-subquery pattern keyed off `ctx`, never args. No work table has a `team` relationship yet, so the helper is generic (a contained cast bridges the relationship type until issue-core lands); it is exercised in tests against `team_membership` (which does expose `team`) to prove scope-vs-deny.
- **Create/insert mutators carry client-minted timestamps in args.** Zero's `InsertValue` requires every non-optional column (reference/zero.md ┬¦5.1 `crud.d.ts`), so `team.create`/`invite.create`/`team.addMember` take `createdAt`/`updatedAt`/`expiresAt` as `number` args minted at the call site (the zbugs `created`/`modified` pattern), alongside the UUIDv7 `id`. The DB `now()` defaults remain the fallback for direct-SQL inserts (`seed`, `acceptInvite`). `invite.createdBy` and every membership's authorization come from `ctx`, never args.

- **Web token wiring (task 5.1) is included; the sign-in and management UI (5.2ŌĆō5.6) is a later phase.** `ZeroRoot` fetches the sync JWT from `/api/zero/token`, hands Zero `userID` + `auth` + optimistic `context`, and re-fetches on the `needs-auth` connection state. With no sign-in UI yet, an unauthenticated visitor gets no token and Zero connects logged-out (server denies) ŌĆö the correct phased state. The e2e sync walking-skeleton test now depends on a session and will pass once the sign-in UI lands; it is not part of the `turbo test` gate.

### UI phase (auth + team-management UI in `apps/web` and `packages/ui`)

- **The access gate decides on the server-resolved role, not the client roster, to avoid a flash.** `ZeroRoot` now exposes a three-state `useSyncSession` (`pending` ŌåÆ first token fetch in flight; `logged-out` ŌåÆ `/api/zero/token` returned non-OK, i.e. no session; `ready` ŌåÆ a 200 carrying the authoritative `role`, which is `null` for an authenticated non-member). `Authenticated` gates on that: `pending` ŌåÆ loading, `logged-out` ŌåÆ redirect to `/login`, `ready` + `role===null` ŌåÆ access gate, `ready` + role ŌåÆ app. Deriving membership from the roster query instead would flash the access gate for a real member during the window where the optimistic `context` is still `undefined` (roster `denyAll`s ŌåÆ empty-complete) before the token arrives. `useMembership` still prefers the *live* roster for role/`memberId` (so role changes and removals reflect immediately), falling back to the token role until the roster settles.
- **A public `GET /api/auth-methods` endpoint reports configured providers (a small, justified server addition in the UI phase).** The login UI must show "Continue with GitHub" only when GitHub is configured and never imply an unconfigured method is paywalled (authentication spec). better-auth exposes no public "enabled providers" endpoint, and the SSO `/sso/providers` list requires a session, so the enabled set has a single source of truth in the server's env. The endpoint returns `{emailPassword:true, github:<configured>, sso:true}`; SSO is always offered because the plugin is always enabled and free (there is no "absent when unconfigured" SSO scenario ŌĆö only GitHub). Verified live against real Postgres: `github:true` with credentials set, `github:false` without.
- **SSO sign-in is an always-available control keyed off the user's email**, calling `authClient.signIn.sso({ email, callbackURL:'/' })` (the `@better-auth/sso` client resolves the provider from the email domain). No static OIDC config exists (Auth-phase decision), and providers are registered at runtime, so enumerating them pre-auth is impossible; surfacing SSO unconditionally matches "OIDC/SSO is available and free" and yields a clear error when no provider matches the domain.
- **Invite acceptance is an inline flow on `/invite?token=`, not a redirect dance.** If the visitor is logged out the invite page renders the `LoginForm` in place; once they sign in/up on the same route, the sync session flips to `ready`, `AcceptInvite` POSTs the token to `POST /api/invites/accept`, calls `useSyncControl().refresh()` to re-mint the token with the new membership, and navigates to `/`. This preserves the token across authentication without threading it through the login redirect, and matches "the invitee interacts through a token, not a synced query" (invitations spec). The invite's accept link is also rendered in a selectable readonly input beside the copy button, satisfying "the link SHALL always be presentable for the admin to copy" without depending on the async Clipboard API.
- **`packages/ui` gained a minimal Base UI primitive set** ŌĆö `Input`, `Label`, `Dialog` (focus-trapping), `Menu` (switcher + user menu), and a `Select` built on a **native `<select>`**. The native select is deliberate: it is fully keyboard-operable and screen-reader-announced with zero composition risk, which fits "functional UI on current styling, do not over-invest in visuals" while guaranteeing the keyboard-first constraint. Role pickers and the add-member picker use it; the workspace/team switcher and user menu use Base UI `Menu` for arrow-key navigation and focus return.
- **The `Label` primitive carries a single `biome-ignore lint/a11y/noLabelWithoutControl`** because it is a generic wrapper whose control association (`htmlFor`) is supplied by each consumer ŌĆö Biome cannot see that statically. Where a control had only an adjacent visible label (member role select, add-member select) the `<label>` wrapper was dropped in favour of an `aria-label` carrying the row's identity, so the accessible name is unique and the rule is satisfied without suppression.
- **`GithubIcon` was replaced with an inline SVG mark** because lucide-react 1.26 removed brand icons; the button keeps its recognizable GitHub glyph without adding a dependency.
- **Web unit tests are kept deterministic by mocking `@/auth/client` and `@/auth/use-auth-methods`** (no jsdom network): `login-form.test.tsx` (5 cases: rendering, provider gating, keyboard submit args, SSO email requirement, sign-up toggle) and the rewritten `routes.test.tsx` (the `/login` route presents the sign-in surface to unauthenticated users).
